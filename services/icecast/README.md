# silvasonic-icecast

> **Status:** Planned (v0.9.0) · **Tier:** 1 (Infrastructure) · **Instances:** Single · **Port:** 8000

**TO-BE:** Icecast acts as the live media streaming server for the Silvasonic platform. It receives low-latency Opus audio streams pushed from Tier 2 Recorders and multiplexes them to one or more listening clients (usually the Web Interface).

---

## The Problem / The Gap

*   **Live Audio:** Scientists and administrators need a way to verify the microphone is working without downloading heavy WAV segments. A real-time listen-in feature is invaluable.
*   **Decoupling:** Recorders cannot handle multiple concurrent student/admin connections. Their job is exclusively data capture. A dedicated streaming server prevents client drops or reconnects from affecting the critical recording loop.

## User Benefit

*   **Real-time Monitoring:** Jump into the "Live" tab on the Web Interface to hear the forest soundscape in near real-time.
*   **Multiple Listeners:** Allows a dozen users to listen to the same microphone simultaneously with zero extra overhead on the Recorder instance or NVMe drive.

---

## Core Responsibilities

*   **Accept Streams:** Opens a dedicated port (e.g., 8000) for authenticated Recorders to push Ogg/Opus streams (the "Source").
*   **Serve Clients:** Multiplexes that stream to any authorized client connecting via HTTP GET requests (the "Listeners").
*   **Mount Points:** Creates dynamic mount points (like `/live-mic-1.opus`) based on the Recorder's input stream name.

---

## Operational Constraints & Rules

| Aspect           | Value / Rule                                                                        |
| ---------------- | ----------------------------------------------------------------------------------- |
| **Immutable**    | Yes — config at startup via `icecast.xml`.                                          |
| **DB Access**    | **No** — Icecast only knows about audio streams and its internal config.            |
| **Concurrency**  | Very High — inherently designed to multiplex one input to many outputs efficiently. |
| **State**        | Stateless — entirely transient data buffered in memory.                             |
| **Privileges**   | Standard (rootless).                                                                |
| **QoS Priority** | `oom_score_adj=0` (default) — Tier 1, but technically non-critical (Life Support).  |

---

## Architecture Context

*   **Upstream:** Tier 2 Recorders act as the audio Source, encoding 16kHz or 48kHz audio to 64kbps Opus streams.
*   **Icecast:** Multiplexes the stream.
*   **Downstream:** The Caddy Gateway routes `/live/*` proxy requests to Icecast. The Web-Interface embeds an `<audio>` HTML tag connected to the Gateway.

---

## Open Questions & Future Ideas

*   **Authentication Validation:** Standard Icecast authentication is simple (source password in config). Can we cleanly integrate this with the gateway or web-interface, or rely on internal network isolation?
*   **Stream Metadata:** Injecting the current frequency or species detections into the Icecast metadata stream (ICY metadata) for real-time display in standard audio players.

## Out of Scope

*   **Does NOT** record audio.
*   **Does NOT** transcode audio (Icecast only forwards the Opus stream generated by the Recorder).
